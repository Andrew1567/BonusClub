## Виконав: Андрій
**Варіант:** 16
**Архітектура:** Hexagonal 

## 1. Опис Use Case
Система дозволяє зареєструвати учасника на наукову конференцію. 
**Основне бізнес-правило:** Реєстрація можлива лише за умови наявності вільних місць (перевірка `max_slots`).

## 2. Обрана архітектура
Було обрано **Гексагональну архітектуру (Ports and Adapters)** для забезпечення незалежності бізнес-логіки від зовнішніх факторів (бази даних чи інтерфейсу).

## 3. Розподіл відповідальностей
Проєкт розділений на три чіткі шари:

1.  **Domain (Бізнес-логіка):**
    - Міститься у `domain/models.py`.
    - Об'єкти `Conference` та `Participant`.
    - **Тут лежить логіка перевірки ліміту місць.** Цей шар не знає ні про які бази даних.
2.  **Application (Сценарії використання):**
    - Міститься у `application/use_cases.py`.
    - Координує потік даних: дістає об'єкт з репозиторію, дає йому команду зареєструвати учасника і зберігає результат.
    - Використовує **Interface (Port)** `ConferenceRepository` для взаємодії із зовнішнім світом.
3.  **Infrastructure (Технічні деталі):**
    - Міститься у `infrastructure/persistence.py`.
    - Реалізує порт у вигляді `InMemoryConferenceRepository`. Це конкретний **Adapter**.

## 4. Чому це Clean Architecture?
- Якщо нам знадобиться замінити зберігання в пам'яті на базу даних MySQL або MongoDB, ми просто створимо новий файл в `infrastructure/`, не змінюючи код у `domain/` чи `application/`.
- Бізнес-правила (ліміт учасників) протестовані окремо від технічних деталей.
